# IoC 容器

## IoC 容器的基本原理是什么？

> 你可以解释一下，Spring 的 IoC 容器如何管理对象的生命周期和依赖关系。比如，什么是“控制反转”？

**面试官**: 那你能解释一下 IoC 容器的基本原理是什么吗？

我：好的面试官，简而言之，IoC（Inversion of Control，控制反转）容器的基本原理就是把对象的创建和管理交给框架来做，而不是让我们手动去创建和维护对象之间的关系。也就是说，通常我们会自己在代码中创建对象，但在 IoC 中，这个“控制权”反转了，交给了容器。

它的核心思想是：**容器负责管理对象的生命周期和它们之间的依赖关系**。你可以理解为，容器就像一个“大管家”，它根据配置（比如 XML 或注解）去创建对象并将依赖注入到这些对象中。

比如说，在传统的开发中，假设有个 `UserService` 类，它依赖一个 `UserRepository` 类来从数据库获取数据。我们通常会在 `UserService` 中通过 `new` 关键字来创建 `UserRepository` 的实例。但在 IoC 容器中，`UserService` 不需要自己去创建 `UserRepository`，它的实例会由容器自动注入。换句话说，容器在幕后做了对象创建、依赖管理这些工作。

**具体来说**，容器通常会通过以下几个步骤来管理 Bean：

1. **实例化对象**：根据配置文件或者注解，容器知道应该创建哪些对象。
2. **依赖注入**：容器根据对象之间的依赖关系，自动注入所需的依赖。可以通过构造方法注入、字段注入等方式。
3. **生命周期管理**：容器不仅负责对象的创建，还会在适当的时候进行初始化和销毁，比如通过 `@PostConstruct` 或 `@PreDestroy` 注解。

总的来说，IoC 容器通过“反转控制”的方式，帮助我们简化了对象创建和管理的过程，让我们能专注于业务逻辑，而不是去关心对象的生命周期和依赖问题。

## Bean 的生命周期如何管理？

> 比如，实例化、依赖注入、初始化和销毁过程如何实现？容器是如何管理这些过程的？

**面试官**：那你能聊聊 Spring 中 Bean 的生命周期是如何管理的吗？

**我**：Bean 的生命周期其实涵盖了从创建、初始化，到销毁的整个过程。Spring 会在每个阶段做一些处理，确保 Bean 能够正确地被管理和使用。

我们从大体流程说起：

1. **实例化**：在 Spring 容器启动时，首先会根据配置（比如 XML 配置文件或者注解）来实例化 `Bean`。通常会根据单例模式来创建 Bean，也就是说，容器只会创建一个 Bean 实例，并且在整个容器生命周期中复用它。对于原型模式的 Bean，每次请求都会创建一个新的实例。
2. **依赖注入**：实例化之后，Spring 会自动完成依赖注入的工作。具体来说，它会通过反射将需要的依赖（比如其他 Bean 或配置的属性）注入到当前 Bean 中。常见的依赖注入方式有构造器注入和字段注入。
3. **初始化**：一旦 Bean 完成了实例化和依赖注入，Spring 会调用一些初始化方法来对 Bean 进行进一步的处理。这个过程包括：
   - 如果 Bean 实现了 `InitializingBean` 接口，Spring 会调用它的 `afterPropertiesSet()` 方法。
   - 如果在 Bean 的配置中指定了初始化方法（通过 `@Bean(initMethod="init")` 或 XML 配置中的 `init-method`），Spring 也会调用这个方法。
4. **使用**：当 Bean 完成了初始化，就可以被容器使用了。也就是，当你通过 `getBean()` 方法获取 Bean 时，容器会直接返回已经完成所有初始化工作的 Bean 实例。
5. **销毁**：Bean 在容器关闭时会被销毁，Spring 会做一些资源清理工作。销毁的步骤包括：
   - 如果 Bean 实现了 `DisposableBean` 接口，Spring 会调用它的 `destroy()` 方法。
   - 如果在 Bean 配置中指定了销毁方法（通过 `@Bean(destroyMethod="destroy")` 或 XML 配置中的 `destroy-method`），Spring 也会调用这个方法。

总结来说，Spring 会在 Bean 生命周期的每个阶段自动管理实例的创建、依赖注入、初始化、使用和销毁，确保对象能够在合适的时机获得正确的处理。

## 单例 Bean 和原型 Bean 有什么区别？如何实现？

> 面试官可能会询问单例 Bean 和原型 Bean 的生命周期管理方式，尤其是在 mini-spring 中是如何实现的。

**面试官**: 那你能讲讲单例 Bean 和原型 Bean 有什么区别吗？它们是如何实现的？

我：单例 Bean 和原型 Bean 是 Spring 中两种常见的 Bean 范式，它们的主要区别体现在**生命周期**和**实例化策略**上。简单来说，单例 Bean 是在整个应用生命周期中只有一个实例，而原型 Bean 每次请求都会创建一个新的实例。

单例 Bean 是 Spring 容器中默认的 Bean 范式。在应用启动时，容器会创建该 Bean 的唯一实例，并且这个实例会被整个容器共享。换句话说，无论你通过多少次 `getBean()` 方法获取这个 Bean，返回的都是同一个实例。

Spring 会在容器启动时，创建并缓存该 Bean 的实例。比如，通过 `@Bean` 注解配置时，默认就是单例模式，或者在 XML 配置中，`scope="singleton"`（这个是默认值）也表示单例。

```java
@Bean
public MyService myService() {
    return new MyService();
}
```

原型 Bean 则不同，每次从容器中请求该 Bean 时，容器都会创建一个新的实例，也就是说，不同的请求会得到不同的 Bean 实例。

你可以通过 `@Scope("prototype")` 注解或者在 XML 配置中设置 `scope="prototype"` 来将 Bean 定义为原型 Bean。

```java
@Scope("prototype")
@Bean
public MyService myService() {
    return new MyService();
}
```

在这种情况下，每次调用 `getBean()` 都会返回不同的实例。

## 容器是如何决定实例化哪个 Bean 的？

> 如果 Bean 需要通过条件注入，容器是如何做选择的？如何实现依赖的匹配？

**面试官**: 那你能讲一下 Spring 容器是如何决定实例化哪个 Bean 吗？

我：Spring 容器决定实例化哪个 Bean，主要是依赖于 Bean 的配置和请求方式。在 Spring 中，Bean 的实例化过程由容器管理，而容器会根据一些配置规则来确定哪些 Bean 需要被实例化以及何时实例化。

1. 基于 Bean 配置：Spring 容器会根据 `BeanDefinition` 中的配置来决定如何实例化 Bean。`BeanDefinition` 描述了一个 Bean 的所有信息，比如类的全名、构造函数参数、属性值等。容器会按照以下几个步骤来决定实例化哪个 Bean：
   1. **通过配置文件或注解配置 Bean**：Spring 会扫描应用的配置文件（如 XML 配置）或者注解（如 `@Bean` 或 `@Component`）来确定有哪些 Bean 需要实例化。
   2. **确定 Bean 的作用域（Scope）**：容器会根据 `scope` 配置决定 Bean 的作用范围。比如，单例 Bean 只会实例化一次，而原型 Bean 每次请求都会实例化一个新的对象。默认情况下，Spring 会实例化单例 Bean。
   3. **依赖注入**：Spring 会检查 Bean 的依赖项（通过构造函数、字段注入或 setter 注入）并决定是否需要实例化它们。如果某个 Bean 依赖于其他 Bean，Spring 会根据依赖关系递归地实例化依赖的 Bean。
2. **懒加载与提前加载（Lazy vs Eager Loading）**：Spring 容器可以通过 `@Lazy` 注解来实现懒加载，这意味着 Bean 只有在真正需要时才会被实例化。如果没有指定懒加载，默认情况下 Bean 会在容器启动时被实例化。
   1. **懒加载（Lazy）**：如果一个 Bean 被标记为懒加载（`@Lazy` 或 `lazy-init="true"`），Spring 不会在容器启动时立即创建该 Bean，而是等到首次使用该 Bean 时才实例化它。
   2. **提前加载（Eager）**：默认情况下，Spring 会在容器启动时实例化所有单例 Bean。
3. **通过 **`**@Autowired**`** 自动装配**：当 Bean 依赖其他 Bean 时，Spring 会自动查找并注入依赖。容器在创建 Bean 时会查找与之相关的依赖项并实例化这些依赖的 Bean。如果是自动装配，Spring 会根据依赖的类型、名称或者构造函数来决定如何实例化和注入 Bean。
4. **容器中的生命周期管理**：Spring 容器会在容器启动时根据 Bean 的定义实例化单例 Bean。如果 `scope="prototype"`，每次请求都会创建新的 Bean 实例。Spring 还会处理对象的初始化和销毁方法。
5. **条件化 Bean 实例化**：
   Spring 还支持条件化 Bean 的实例化。例如，你可以使用 `@Conditional` 注解，或者在 XML 配置中使用 `<bean>` 标签的 `lazy-init` 属性，来基于某些条件决定是否实例化 Bean。比如，某些 Bean 可能只有在特定条件满足时才会被实例化。

# AOP

## AOP 的核心原理是什么？

> 你可以介绍 AOP 的基本概念，如连接点（JoinPoint）、切点（Pointcut）、通知（Advice）和增强（Aspect）

**面试官**: 那你能讲讲 AOP 的核心原理是什么？

我：AOP（Aspect-Oriented Programming，面向切面编程）的核心原理主要是通过将横切关注点从核心业务逻辑中分离出来，来增强程序的模块化性。在传统的面向对象编程中，功能通常是通过类和对象的继承和多态来实现的，但很多时候，某些功能（比如日志记录、安全验证、事务管理等）是横跨多个模块的，这些功能通常会被称为“横切关注点”，它们会影响到多个类或方法。

AOP 就是通过对这些横切关注点的抽象，使得我们能够将它们独立出来，以减少冗余代码，提升模块化和可维护性。

**AOP 核心概念：**

+ **切面（Aspect）**：切面是 AOP 的核心概念之一，它是对横切关注点的模块化。比如，一个日志记录的功能就可以看作一个切面，它会应用到多个类和方法上。切面由两部分组成：**通知（Advice）** 和 **切点（Pointcut）**。
+ **通知（Advice）**：通知定义了具体的增强行为。它描述了在何时、何地对方法进行增强。通知有多种类型，常见的有：
+ **前置通知（Before）**：在目标方法执行之前执行。
+ **后置通知（After）**：在目标方法执行之后执行。
+ **环绕通知（Around）**：包裹目标方法执行的通知，可以在执行前后都做处理。
+ **返回通知（AfterReturning）**：在目标方法正常返回后执行。
+ **异常通知（AfterThrowing）**：在目标方法抛出异常后执行。
+ **切点（Pointcut）**：切点定义了哪些方法会被通知增强，它通过表达式来描述要匹配的方法。Spring AOP 使用切点表达式（如 `execution(* com.example.service.*.*(..))`）来指定哪些方法应该被增强。
+ **连接点（JoinPoint）**：连接点表示程序执行过程中的某个点，通常是方法调用。AOP 会在这些连接点插入通知来增强目标方法的行为。比如，在方法执行前、执行后或者抛出异常时。
+ **代理（Proxy）**：AOP 通过创建代理对象来拦截对目标方法的调用，这就是实现 AOP 的关键。Spring 中有两种代理方式：
+ **JDK 动态代理**：当目标对象实现了接口时，Spring 会基于接口生成代理对象。
+ **CGLIB 代理**：当目标对象没有实现接口时，Spring 会通过 CGLIB 动态生成目标对象的子类来进行代理。

## 如何实现 AOP 代理？

> Spring 是如何通过 JDK 动态代理或 CGLIB 代理来实现 AOP 的？你可以解释一下你的 mini-spring 是如何创建代理对象并应用通知的。

**面试官**: 那你能讲讲如何实现 AOP 代理吗？

我：实现 AOP 代理的核心是使用代理机制来拦截目标方法的执行，然后在方法执行前、后或替代执行的地方加入额外的逻辑。Spring 中的 AOP 代理通常有两种实现方式：**JDK 动态代理**和**CGLIB 代理**。我会分别解释这两种方式。

JDK 动态代理是基于 Java 的反射机制实现的，它创建一个实现了指定接口的代理类。JDK 动态代理只能代理那些实现了接口的类。它的实现步骤如下：

** 1、定义目标对象和接口**

目标对象必须实现一个接口，代理对象会实现这个接口，所有的方法调用会转发给代理对象。

```java
public interface UserService {
    void registerUser(String userName);
}

public class UserServiceImpl implements UserService {
    public void registerUser(String userName) {
        System.out.println("User " + userName + " registered.");
    }
}
```

2、**定义 InvocationHandler**

`InvocationHandler` 是 JDK 动态代理的核心接口，它定义了代理对象的行为。当目标方法被调用时，`invoke()` 方法会被触发。

```java
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;

public class LogInvocationHandler implements InvocationHandler {
    private Object target;

    public LogInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 在方法执行前增强逻辑
        System.out.println("Logging before calling method: " + method.getName());
        Object result = method.invoke(target, args);  // 调用目标方法
        // 在方法执行后增强逻辑
        System.out.println("Logging after calling method: " + method.getName());
        return result;
    }
}
```

**3、生成代理对象**

使用 `Proxy.newProxyInstance()` 方法来生成代理对象，并通过 `InvocationHandler` 来拦截方法调用。

```java
import java.lang.reflect.Proxy;

public class AopDemo {
    public static void main(String[] args) {
        UserService userService = new UserServiceImpl();
        InvocationHandler handler = new LogInvocationHandler(userService);

        // 创建代理对象
        UserService proxy = (UserService) Proxy.newProxyInstance(
            UserService.class.getClassLoader(),
            new Class<?>[]{UserService.class},
            handler
        );

        // 调用代理方法
        proxy.registerUser("Alice");
    }
}
```

CGLIB（Code Generation Library）是一个功能强大的字节码生成库，它可以在运行时动态生成目标类的子类，并在子类的方法中插入代理逻辑。CGLIB 代理不要求目标对象实现接口，因此可以代理没有实现接口的类。

**1、目标类定义**

目标类不需要实现接口，可以是一个普通的类。

```java
public class UserService {
    public void registerUser(String userName) {
        System.out.println("User " + userName + " registered.");
    }
}
```

**2、定义 MethodInterceptor**

CGLIB 使用 `MethodInterceptor` 来拦截方法调用。在 `intercept()` 方法中，你可以插入自定义的增强逻辑。

```java
import net.sf.cglib.proxy.Enhancer;
import net.sf.cglib.proxy.MethodInterceptor;
import net.sf.cglib.proxy.MethodProxy;

public class LogMethodInterceptor implements MethodInterceptor {
    private Object target;

    public LogMethodInterceptor(Object target) {
        this.target = target;
    }

    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        // 在方法执行前增强逻辑
        System.out.println("Logging before calling method: " + method.getName());
        Object result = method.invoke(target, args);  // 调用目标方法
        // 在方法执行后增强逻辑
        System.out.println("Logging after calling method: " + method.getName());
        return result;
    }
}
```

**3、生成代理对象**

使用 CGLIB 的 `Enhancer` 类来创建目标类的子类，并在该子类的方法中执行增强逻辑。

```java
public class AopDemo {
    public static void main(String[] args) {
        UserService userService = new UserService();

        // 创建 CGLIB 代理
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(UserService.class);
        enhancer.setCallback(new LogMethodInterceptor(userService));

        // 创建代理对象
        UserService proxy = (UserService) enhancer.create();

        // 调用代理方法
        proxy.registerUser("Bob");
    }
}
```

Spring 的实现是：

如果目标对象实现了接口，使用 JDK 动态代理，目标对象没有实现任何接口，CGLIB 通过子类化目标对象来进行代理。

## 什么是切点表达式？如何匹配目标方法？

> 你可以解释 Spring 中的切点表达式是如何定义的，以及它如何匹配目标方法，并触发相应的通知。

**面试官**: 那你能讲一下什么是切点表达式吗？它是如何匹配目标方法的？

我：切点表达式是 AOP 中的一个非常重要的概念，它决定了哪些方法会被拦截并应用通知。我们可以通过切点表达式来定义一组规则，用于匹配目标方法。Spring AOP 使用切点表达式来指定哪些方法需要增强（比如执行日志、事务处理等横切关注点）。

切点表达式是一个用来匹配连接点（方法调用）的表达式，它描述了拦截哪些方法。通过切点表达式，Spring AOP 能够根据方法签名、类名、包路径等特征来匹配目标方法。

切点表达式使用一种类似于正则表达式的语法，称为 **Pointcut 表达式语言**，它支持在方法上使用 `execution`、`within`、`args` 等关键字来描述目标方法。

`**execution**`** 表达式**：

`execution` 是最常用的切点表达式，它用来匹配目标方法的执行。它的基本格式是：

```java
execution(modifiers-pattern? return-type-pattern declaring-type-pattern? method-name-pattern(param-pattern) throws-pattern?)
```

+ **modifiers-pattern**：方法修饰符，通常是 public、private、protected 等（可选）
+ **return-type-pattern**：方法的返回类型
+ **declaring-type-pattern**：目标方法所在的类或接口
+ **method-name-pattern**：方法名称
+ **param-pattern**：方法的参数类型
+ **throws-pattern**：方法抛出的异常类型（可选）

切点表达式不仅可以用于定义通知要增强哪些方法，还可以根据业务需要灵活配置。例如：

+ **日志记录**：通过切点表达式匹配业务方法来记录日志。
+ **事务管理**：通过切点表达式匹配需要事务管理的方法，在方法执行前后进行事务的启动和提交/回滚操作。
+ **权限校验**：通过切点表达式匹配需要权限验证的方法，在执行前校验用户权限。

## AOP 中的通知（Advice）有哪几种类型？

> 讨论 Spring 中常见的通知类型（前置通知、后置通知、环绕通知、异常通知等），以及你是如何在 mini-spring 中实现这些功能的。

**面试官**: 那你能讲一下 AOP 中的通知（Advice）有哪几种类型吗？

我：在 AOP 中，**通知（Advice）** 是指在特定连接点执行的增强逻辑。它决定了目标方法的前后或者替代执行的行为。Spring AOP 中的通知有几种不同的类型，每种通知类型对应着不同的执行时机和目的。

1. 前置通知前置通知是在目标方法执行之前执行的通知。它通常用于执行一些预处理操作，比如日志记录、权限检查、参数验证等。
   1. **执行时机**：在目标方法调用之前执行。
   2. **是否能阻止方法执行**：前置通知是无法阻止目标方法的执行的。如果需要阻止方法执行，可以通过抛出异常来实现。
2. 后置通知（After Advice）后置通知是在目标方法执行之后执行的通知。不管目标方法是否抛出异常，后置通知都会执行。通常用于清理资源或记录执行结果。
   1. **执行时机**：目标方法执行之后，不管方法是否抛出异常
   2. **是否能阻止方法执行**：后置通知无法阻止目标方法的执行
3. 返回通知（AfterReturning Advice）返回通知是在目标方法执行成功并返回结果之后执行的通知。它可以访问到目标方法的返回值，并且可以基于返回值做进一步的处理。
   1. **执行时机**：目标方法执行成功并返回结果之后
   2. **是否能阻止方法执行**：返回通知只能在目标方法执行后被调用，不能影响目标方法的返回值
4. 异常通知（AfterThrowing Advice）异常通知在目标方法抛出异常之后执行，通常用于异常处理，比如日志记录、异常的补偿操作或事务的回滚。
   1. **执行时机**：目标方法抛出异常之后。
   2. **是否能阻止方法执行**：异常通知不会影响目标方法的正常执行，只在发生异常时触发。
5. 环绕通知（Around Advice）环绕通知是最强大也是最灵活的一种通知类型，它能够完全控制目标方法的执行。通过环绕通知，我们可以在方法执行前后加入自定义逻辑，甚至决定是否执行目标方法（通过 `proceed()` 方法）。
   1. **执行时机**：环绕通知会在目标方法执行之前和执行之后都可以执行。它通过 `joinPoint.proceed()` 来调用目标方法，并且能够控制目标方法是否执行，如何执行。
   2. **是否能阻止方法执行**：可以阻止目标方法执行，甚至替代目标方法的执行结果。

## AOP 与装饰者模式的区别是什么？

> 装饰者模式和 AOP 都是用来扩展功能的，但它们的实现方式、应用场景等方面有所不同，面试官可能会要求你解释这两者的区别。

**面试官**: 那你能讲一下 AOP 与装饰者模式的区别是什么吗？

我：AOP（面向切面编程）和装饰者模式（Decorator Pattern）看起来有些相似，都是用来增强对象的功能，但它们的目的、实现方式和应用场景有很大的不同。让我通过几个方面来详细解释它们的区别。

1. 目的
   1. **AOP（面向切面编程）**：AOP 的目的是将横切关注点（比如日志、事务、安全等）从核心业务逻辑中分离出来。AOP 主要关心的是如何在运行时动态地插入增强行为，而不修改目标对象的代码。AOP 适合于在多个不同模块中共享某些功能（如日志记录、事务管理等）。
   2. **装饰者模式**：装饰者模式的目的是动态地扩展对象的功能，通常是在不修改对象类的情况下，向现有对象添加新的行为。装饰者模式通过创建“装饰者”类，包裹原始对象，在不改变对象本身的情况下增强它的行为。它通常用于给单个对象添加功能。
2. 实现方式不同
   1. AOP 通过代理机制来实现对目标对象的增强，代理对象会在目标方法执行前、执行后或者替代执行时插入增强逻辑。Spring AOP 使用动态代理技术（如 JDK 动态代理或 CGLIB）来为目标对象创建代理，切点和通知决定了增强的时机和行为。AOP 可以通过配置文件或注解方式来配置。
   2. 装饰者模式通过构建一系列装饰者类来包装原始对象，每个装饰者都提供额外的功能。装饰者类和被装饰的类实现相同的接口，通过继承或实现相同的接口来确保它们能够互相替代。装饰者模式通常通过手动创建装饰对象来实现，且需要通过代码逻辑层层包裹。
3. 应用场景不同
   1. AOP 主要用于 **横切关注点** 的处理，如日志、事务管理、性能监控、安全检查等，这些功能通常会影响多个模块的多个方法，适合抽象成一个切面来处理，避免在每个方法中重复代码。AOP 适用于处理跨多个类或方法的逻辑增强。
   2. 装饰者模式更适合用于给单个对象动态添加额外功能，而不会影响对象的其它部分。它通常用于单个对象的功能扩展，如对输入流、输出流的包装、窗口控件的样式和功能增强等。装饰者模式适合对具体对象进行层次化扩展。
4. 增强位置不同
   1. AOP 是通过 **代理对象** 来增强目标对象的行为，目标对象的代码并没有直接被修改。增强逻辑是在运行时通过代理机制插入的，这意味着 AOP 关注的是对象的“行为”层面。
   2. 装饰者模式是通过 **包装对象** 来增强功能，装饰者类会包裹原始对象，并通过重写接口中的方法来为对象提供新的功能。装饰者模式关注的是对象的“结构”层面，装饰者通过继承或组合的方式扩展对象的行为。
5. 松耦合与灵活性
   1. AOP 通过代理机制和切点表达式来进行增强，因此它具有较高的 **松耦合性**。AOP 不需要直接修改目标对象的代码，而且增强的逻辑可以集中管理，增强功能可以灵活地在多个方法或类中复用。
   2. 装饰者模式通过组合和继承来实现功能增强，相比 AOP，它的 **耦合度** 相对较高，因为装饰者需要与目标对象共享相同的接口，且每次增强都需要创建一个新的装饰类。在某些情况下，装饰者链可能会变得较为复杂和笨重。
6. 是否影响目标对象的类
   1. AOP 不修改目标对象的类，它是通过代理机制在运行时对目标对象的方法进行增强，目标对象本身不需要做任何改动。
   2. 装饰者模式会通过创建一个装饰类来增强目标对象，它通常会直接修改目标对象的行为或状态，但并不会修改目标对象的原始类。

## 动态代理和静态代理的区别是什么？

> 你可以谈谈 JDK 动态代理和静态代理的实现原理，以及它们的优缺点，特别是在 AOP 实现中的应用。

**面试官**: 那你能讲一下动态代理和静态代理的区别吗？

**我**: 动态代理和静态代理都是代理模式的一种实现，它们的目标都是通过代理对象来控制对目标对象的访问，在不改变目标对象代码的情况下，增强目标对象的功能。但是它们在实现方式、灵活性和使用场景上有很大的区别。

**1、定义**

+ **静态代理**：静态代理是通过手动创建代理类来增强目标对象的功能。代理类在编译时就已经确定好了，代理对象和目标对象的类都已经存在，并且在代码中明确地定义了代理方法。静态代理通常是在编译时就生成的，代理类需要手动编写。
+ **动态代理**：动态代理是在运行时创建代理对象，并且不需要显式地编写代理类。动态代理通常使用反射机制来创建代理对象，在执行期间动态地将增强逻辑织入到目标方法中。Java 中的动态代理通过 `Proxy` 类和 `InvocationHandler` 接口实现。

**2、代理类的生成方式**

+ **静态代理**：代理类是在编译时就确定的，通常由开发者手动编写或者通过代码生成工具生成。每个需要代理的类都需要编写一个代理类，代理类会实现与目标对象相同的接口或继承相同的父类，并在其中提供对目标方法的调用和增强。
+ **动态代理**：代理类是在运行时动态生成的，通常由代理框架（如 JDK 动态代理、CGLIB）生成。在 Java 中，JDK 动态代理通过 `Proxy.newProxyInstance()` 方法创建代理类，CGLIB 通过字节码生成技术动态生成目标类的子类。

**3、灵活性**

+ **静态代理**：静态代理的灵活性较低。每次需要代理一个新的类时，都必须手动编写代理类。如果目标类有多个，代理类的数量会呈指数增长，而且需要在每个代理类中重复实现相同的逻辑（如调用目标方法）。
+ **动态代理**：动态代理的灵活性非常高。通过动态代理，可以在运行时指定代理对象，而无需手动编写代理类。只需要提供一个 `InvocationHandler`，然后就可以为任意目标对象生成代理，适应性和扩展性都很强。

4. **代码示例**

+ **静态代理示例**：

```java
// 目标接口
public interface UserService {
    void registerUser(String username);
}

// 目标类
public class UserServiceImpl implements UserService {
    @Override
    public void registerUser(String username) {
        System.out.println("User " + username + " registered.");
    }
}

// 静态代理类
public class UserServiceProxy implements UserService {
    private UserServiceImpl userService;

    public UserServiceProxy(UserServiceImpl userService) {
        this.userService = userService;
    }

    @Override
    public void registerUser(String username) {
        System.out.println("Logging before user registration...");
        userService.registerUser(username);
        System.out.println("Logging after user registration...");
    }
}

// 使用代理
public class Main {
    public static void main(String[] args) {
        UserService userService = new UserServiceImpl();
        UserService proxy = new UserServiceProxy(userService);
        proxy.registerUser("Alice");
    }
}
```

在静态代理中，`UserServiceProxy` 类需要手动编写，并且每个需要代理的类都需要一个对应的代理类。

+ **动态代理示例（JDK）**：

```java
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

// 目标接口
public interface UserService {
    void registerUser(String username);
}

// 目标类
public class UserServiceImpl implements UserService {
    @Override
    public void registerUser(String username) {
        System.out.println("User " + username + " registered.");
    }
}

// 动态代理类
public class LogInvocationHandler implements InvocationHandler {
    private Object target;

    public LogInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("Logging before method execution...");
        Object result = method.invoke(target, args); // 调用目标方法
        System.out.println("Logging after method execution...");
        return result;
    }
}

// 使用代理
public class Main {
    public static void main(String[] args) {
        UserService userService = new UserServiceImpl();
        InvocationHandler handler = new LogInvocationHandler(userService);

        // 创建代理对象
        UserService proxy = (UserService) Proxy.newProxyInstance(
            UserService.class.getClassLoader(),
            new Class<?>[]{UserService.class},
            handler
        );

        proxy.registerUser("Bob");
    }
}
```

在动态代理中，`LogInvocationHandler` 处理目标方法的增强，代理类是动态生成的。

5. **性能**

+ **静态代理**：静态代理的性能较好，因为代理类在编译时已经生成，不需要进行反射调用。每次调用方法时，直接调用代理类中的方法即可。
+ **动态代理**：动态代理的性能较差，因为它涉及到反射机制，反射的调用效率相对较低，尤其是在方法调用频繁的情况下。不过，现代 JVM 对反射的优化已经使得这种性能差异变得不那么明显。

6. **使用场景**

+ **静态代理**：适用于目标类和代理类明确且固定的情况，或者目标类较少时。静态代理的实现比较简单，但当代理类数量庞大时，维护会比较困难。
+ **动态代理**：适用于目标类较多、无法提前确定代理对象的场景。动态代理非常灵活，可以在运行时创建代理对象，而无需修改目标对象的代码。常用于 Spring AOP 等框架中，能够为多个目标类提供统一的代理增强功能。
